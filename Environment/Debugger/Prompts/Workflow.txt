
DEBUGGING WORKFLOW

We need to debug the '[[project]]' project so it can successfully pass the provided test cases. Although the task is complex, we will approach it in a structured, step-by-step manner.

Steps for each test case
1 Run Test Commands
 - The system executes all commands related to the test: pre-test commands, the main test command, and post-test commands.
 - All commands produce and consume files only from the working directory.
2 System analysis - analyze logs/trace/source for functions qualified as sub-systems.
 - Review output logs (including trace logs) to determine if the test passed.
 - If it has passed successfully, we stop here.
 - If the test fails identify the sub-system and next high-priority blocker.
3 Gather Additional Information
 - If the test did not pass, collect relevant details such as suspected functions, data types, error messages or debug information provided by the trace and log. Try to understand the big picture.
 - Identify symptoms of the next blocker. Determine which function is failing from logs and traces
 - Locate the issue. Find the problematic code in that function or identify corrupted input
 - Trace root cause if the issue stems from bad input:
   * Follow the data flow up the call stack
   * Inspect logs and traces for upstream function calls
 - Use debug_function action if logs/traces are insufficient
   * Add breakpoints in suspected functions
   * Set appropriate conditions and watch expressions to verify code execution paths not represented in current logs/traces
4 Determine the Next Step
 - Based on the collected information, decide which action to take next and assemble any required data or context.
5 Perform the Chosen Action
 - Summarize all important information in the 'Debug notes' and 'motivation' sections. Anything else might not be available further in the next steps of the process.
 - Build a complete hypothesis of the problem's root cause and the solution, then proceed to the fix.
6 Fix - Once root cause is identified:
 - Choose the fix that best addresses the issue. Considering the tradeoff "Structural vs. Local" changes based on which is likely to reach a passing test in the fewest steps. 
 - Avoid introducing regressions or reward-hacking
7 Verify the Fix
 - After each fix_function action the system will automatically start the tests. Confirm the fix after each run
8 Repeat the above steps (from 2) if the test still fails

Maintaining detailed “Debug Notes”
Throughout this process, maintain a Debug Notes document, updating it after each step. The Debug Notes should capture:
 - Completed Actions: What has been done so far.
 - Current Status: Where we are in the debugging process.
 - Plan & Goals: Short-term, mid-term, and long-term objectives.
 - Next Steps: Which actions we intend to take and why.
The Debug Notes serve as a quick reference to guide decisions at each phase; update them at the end of every step.

Possible Next Actions. 
When deciding on the next action (in Step 4 above), you can choose an action from the following groups of actions:
1 Actins for requesting more information
 - log_info: Requests a section from the application log starting from a given log line number. Optionally filters by function name and 1-based function invocation. Returns a length-limited log section.
 - function_info: Requests information for a function, including source code, description, log and trace events. The log entries and trace events are for the specified 1-based function invocation, defaults to first invocation if unspecified.
 - data_info: Request information for a data type (struct or enum), including the source of the definition
 - file_info: Request information for a file (cat file from the working directory of the test). 1-based line number from where to start reading the file could be provided. There might be a limit applied to the maximum character red and displayed from the file.
 - call_graph: ASCII graph displaying the function call hierarchy, showing which functions call other functions. Helps visualize code execution paths and dependencies. By default starts from the application entry point but root node can be specified optionally as a subject to this action. Printed depth of the graph is a subject to limitation. 
 - functions_summary: Request brief descriptions (1-2 sentences) and declarations for functions defined by the application. By default prints hierarchically all functions, starting from the entry point but root function can be specified. Printed list is subject to limitation based on the call stack depth.
 - step_info: Shows summary of the trajectory of the debugging progress the way it used to be at the step with requested index. Could be useful when tracking regressions or trying to understand more about debugging in the past.
 - search_source: Scans in the data type and function source files for lines matching a give regex pattern. Don't use search_source to view sources - for that use function_info and data_info actions
2 Actions for changing the source if there is enough information what the problem is and how to fix it.
 - fix_function: Fix an existing function, without changing function signature and used data types. Fixing the function might involve changing its source, calling existing functions or introducing new helper functions.
 - refactor_data: Refactor an existing data type by changing existing struct or enum members. Note, refactoring existing data types can affect many parts of the codebase, so proceed only when necessary.
 - new_data: Introduce a new data type or new members to an existing struct or enum. Most probably, as a next step, we will have to introduce new functions or change existing one to use the new data type
3 Actions for debugging, verifying application behavior, validate fixes and collect logs and traces
 - debug_function: Debug a function via the debugger in batch mode by setting trace points on entering and exiting the debugged function while running the test. Each trace point might provide information for further analysis about the call stack, source location and live variables at that location. Note that each trace point can be hit maximum [[hit_count]] times and after that will be deactivated. Those trace points will be set automatically by a setup script. Optionally, set custom breakpoints as described in the section 'HOW TO SET BREAKPOINTS'. When breakpoints are hit, traces and logs from the resulting call stacks, for each function and invocation, can be requested using the 'function_info' action. This data remains available until the next test run.
 - run_test: Re-run the test. Trace and log from the application execution will be collected for further analysis.
