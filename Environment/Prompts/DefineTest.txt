[[recommendation_note]]
I prefer a simple test that guarantees sub-systems contract with a goal of passing the full application test. We can implement a test that covers more later.

The unit test code must assume PRINT_TEST macro is already defined and to not redefine it.

Important characteristic of the unit test is that the test itself resides in single main.cpp file (the test driver), wrapping the tested function. All test cases and setup need to be in that file (main.cpp). This file will be compiled and the executable will be executed with the arguments provided in the command from the 'test' step. The purpose of the 'pretest' is usually to generate any intermediate input files needed for the 'test' step - consumed by the test executable. The 'posttest' step is usually to post-process the output files from the test executable and to report errors. The 'pretest' and 'posttest' steps aren't required and should be used only if needed to support the main test executable.

 The unit tested application function '[[function]]' and all data types for its arguments and return type are already implemented and the test doesn't have to implement them. Other application functions won't be visible for the unit test and shouldn't be used. The test script must not generate main.cpp or to compile the executable being tested. The unit test executable must not be executed in the pretest and posttest steps. However, the framework will invoke the executable with the arguments specified in the 'test.command' field.

Please, note I'm not asking you here to implement the test's main.cpp. I'm not asking to provide the content of the input files. I will ask for the source of the test and content of the input files (if any) later in a separate step. I'm asking you only to provide description, information about the expected command line and input files as discussed above.
