REVIEW UNIT TEST

Debugging of unit test '[[old_test_name]]' for function '[[function]]' has concluded with failure.

Analyze the DEBUGGING TRAJECTORY to determine whether the failure stems from the unit test itself or from the application code. Since both the application and the test were generated by an LLM, we must determine whether:

The unit test is incorrect and needs improvement, or
The failure is purely on the application side and requires further debugging

Only blame the unit test if you can cite specific evidence from the debugging trajectory that demonstrates the test itself is faulty. Do not assume test issues without concrete indicators.
Common reasons for a unit test to cause be incorrect:

A bug in the test driver (main.cpp) and test input files and this is the reason unit test conditions can't be satisfied.  
Missing or incorrect input files. 
Input source files for the test steps, intended as valid, that contain syntax errors — including unclosed or malformed comments.
Incorrect command-line arguments in pretest/test/posttest commands
Wrong expected values (especially floating-point comparisons, edge cases)
Wrong assertions (testing the wrong property)
Missing setup/teardown logic
Hardcoded paths or environment-specific assumptions
Matching stdout in tests with PRINT_TEST. Only std::cout is captured for stdout matching. PRINT_TEST is only for debug log and must not be used to match the regex pattern provided in the 'stdout' in the test specification (see test framework manual)
PRINT_TEST is a system level macro and can't be changed!

NOT A REASON TO BLAME THE TEST:
- It's hard to satisfy
- The application keeps failing it
- The test is doesn't have coverage on a certain features
- The test is reward-hackable

Note:
Stdout regex is checked with C++ std::regex ECMAScript using std::regex_match (FULL string) after normalizing newlines to \n; . does not match \n, so .* won’t cross lines or consume a trailing newline—use [\s\S]* (or \n? at the end).
Example: log TEST: SUCCESS\n → use ^.*TEST: SUCCESS.*\n?$ (not ^.*TEST: SUCCESS.*$).

If the unit test needs improvement, provide a detailed diagnosis explaining:

What specific evidence from the trajectory indicates the test is incorrect
What exactly is wrong with the current test
How the corrected test should behave differently

This feedback will be used to regenerate the test from scratch. All debugging progress will be reverted to ensure an optimal trajectory.
